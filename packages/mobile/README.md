# @actual-app/mobile

Native iOS and Android apps for Actual Budget. These are
[Capacitor](https://capacitorjs.com/) wrappers around the existing web build
(`@actual-app/web`): the React web UI and the loot-core Web Worker backend run
unchanged inside the platform WebView, and the SQLite database persists into
IndexedDB exactly as it does in a browser.

There is **no separate UI codebase** — everything ships from `packages/desktop-client`.

## Prerequisites

Capacitor needs platform tooling that `yarn` does not install:

- **iOS** (macOS only): Xcode (iOS SDK + Simulator) and CocoaPods.
- **Android**: JDK 17 and the Android SDK (Android Studio recommended).
- An Apple Developer account / Google Play developer account for publishing.

## How it fits together

```
packages/desktop-client/build      ← web build (vite --mode=capacitor)
        │  bin/stage-webdir.mjs
        ▼
packages/mobile/webdir              ← disposable copy Capacitor reads (gitignored)
        │  cap sync <platform>
        ▼
packages/mobile/ios/App             ← native Xcode project
packages/mobile/android             ← native Gradle project
```

- The web build is produced in **`capacitor` mode** (`vite --mode=capacitor`),
  which keeps absolute (`/`) asset paths — Capacitor serves the bundle from the
  WebView's local origin — and disables the service worker.
- `bin/stage-webdir.mjs` copies the build into `webdir`.
- `cap sync <platform>` copies `webdir` into the native project.

## First-time setup (run once)

The native `ios/` and `android/` projects are generated by Capacitor and are
**not** fully checked in by the agent that scaffolded this package. Generate
them and wire the per-platform cross-origin-isolation hook:

```bash
# from the repo root
yarn install
yarn workspace @actual-app/mobile build:web   # produces desktop-client/build
yarn workspace @actual-app/mobile copy-web     # stages webdir

# iOS (macOS):
yarn workspace @actual-app/mobile add:ios      # cap add ios
yarn workspace @actual-app/mobile sync:ios

# Android:
yarn workspace @actual-app/mobile add:android  # cap add android
yarn workspace @actual-app/mobile sync:android
```

### Enabling SharedArrayBuffer (cross-origin isolation)

loot-core's sql.js/absurd-sql backend is fastest when `SharedArrayBuffer` is
available, which requires the WebView to be cross-origin isolated (COOP/COEP
response headers). Each platform has a small native injector already in this
package:

- **iOS**: `ios/App/App/COEPSchemeHandler.swift` — add it to the `App` target,
  add `WKAppBoundDomains` (`localhost`) to `Info.plist`, and wire it per the
  doc comment at the top of the file.
- **Android**: `android/app/src/main/java/org/actualbudget/app/COEPBridgeWebViewClient.kt`
  — set it as the WebView client in `MainActivity` (snippet in the file's doc
  comment).

> **If you skip the native hook:** the app still works. The web layer detects
> the missing isolation and automatically switches to the slower
> no-`SharedArrayBuffer` mode (`browser-preload.js`). On mobile there is only a
> single WebView, so that fallback is safe.

### Icons & splash

Generate the asset catalogs for both platforms from a 1024×1024 source:

```bash
npx @capacitor/assets generate \
  --assetPath ../desktop-client/public \
  --iconBackgroundColor '#5c3dbb' --splashBackgroundColor '#5c3dbb'
```

(Provide a 1024px logo; the brand assets in `desktop-client/public` are 512px.)

## Daily development

```bash
# from the repo root: build web + stage + cap sync
yarn build:ios          # or: yarn build:android

# open the native project in Xcode / Android Studio
yarn start:ios          # or: yarn start:android

# or run straight onto a simulator/emulator/device
yarn workspace @actual-app/mobile run:ios       # or run:android
```

Verify in the WebView inspector (Safari ▸ Develop for iOS; `chrome://inspect`
for Android):

- `self.crossOriginIsolated === true` and `typeof SharedArrayBuffer === 'function'`
  (only when the native COEP hook is wired; otherwise expect the fallback)
- the WASM module loads with `Content-Type: application/wasm`
- no CSP violations, a budget can be created, and data survives a relaunch

## Store pipelines

CI runs manual (`workflow_dispatch`) lanes:

- `.github/workflows/ios.yml` → TestFlight / App Store (`macos-latest`)
- `.github/workflows/android.yml` → Google Play (`ubuntu-latest`)

Each uses Fastlane (`ios/App/fastlane/`, `android/fastlane/`) with a `beta`
(TestFlight / Play internal track) and a `release` lane.

```bash
# iOS
cd packages/mobile/ios/App && bundle install && bundle exec fastlane beta
# Android
cd packages/mobile/android && bundle install && bundle exec fastlane beta
```

### Required secrets / one-time human steps

These need an Apple Developer / Google Play account and **cannot** be done by an
automated agent.

**iOS** (GitHub `release` env):

| Secret                                           | What it is                                                          |
| ------------------------------------------------ | ------------------------------------------------------------------- |
| `APPLE_TEAM_ID`                                  | Apple Developer Team ID                                             |
| `ASC_KEY_ID`, `ASC_ISSUER_ID`, `ASC_KEY_CONTENT` | App Store Connect API key (`ASC_KEY_CONTENT` = base64 of the `.p8`) |
| `MATCH_GIT_URL`                                  | private git repo holding signing certs/profiles                     |
| `MATCH_PASSWORD`                                 | match encryption passphrase                                         |
| `MATCH_GIT_BASIC_AUTHORIZATION`                  | base64 `user:token` with read access to the match repo              |

One-time: register the bundle id `org.actualbudget.app` + create the App Store
Connect app record, then seed signing assets with
`cd ios/App && bundle exec fastlane match appstore`.

**Android** (GitHub `release` env):

| Secret                                                                   | What it is                              |
| ------------------------------------------------------------------------ | --------------------------------------- |
| `ANDROID_KEYSTORE_BASE64`                                                | base64 of the upload keystore (`.jks`)  |
| `ANDROID_KEYSTORE_PASSWORD`, `ANDROID_KEY_ALIAS`, `ANDROID_KEY_PASSWORD` | keystore credentials                    |
| `PLAY_STORE_JSON_KEY_BASE64`                                             | base64 of the Play service-account JSON |

One-time: create the app in the Play Console, create an upload keystore, and a
service account with "Release manager" access. `app/build.gradle` must read the
`android.injected.signing.*` properties Fastlane passes (this is Gradle's
built-in injected-signing support).

Both Fastfiles read the version name from this package's `package.json`
`version`, so keep it in lockstep with `@actual-app/web`. The build/version
code comes from the CI run number.
