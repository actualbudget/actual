# @actual-app/ios

Native iOS app for Actual Budget. This is a [Capacitor](https://capacitorjs.com/)
wrapper around the existing web build (`@actual-app/web`): the React web UI and
the loot-core Web Worker backend run unchanged inside a `WKWebView`, and the
SQLite database persists into IndexedDB exactly as it does in a browser.

There is **no separate UI codebase** — everything ships from `packages/desktop-client`.

## Prerequisites

These cannot be installed by `yarn` and must exist on a **macOS** machine:

- Xcode (with the iOS SDK and Simulator)
- CocoaPods (`gem install cocoapods`, or via the bundled `Gemfile`)
- An Apple Developer account (for signing / App Store, not for the Simulator)

## How it fits together

```
packages/desktop-client/build      ← web build (vite --mode=capacitor)
        │  bin/stage-webdir.mjs
        ▼
packages/desktop-ios/webdir         ← disposable copy Capacitor reads (gitignored)
        │  cap sync ios
        ▼
packages/desktop-ios/ios/App        ← native Xcode project
```

- The web build is produced in **`capacitor` mode** (`vite --mode=capacitor`),
  which keeps absolute (`/`) asset paths — Capacitor serves the bundle from the
  custom-scheme root — and disables the service worker.
- `bin/stage-webdir.mjs` copies the build into `webdir`.
- `cap sync ios` copies `webdir` into the native project and installs pods.

## First-time setup (run once, on macOS)

The `ios/` Xcode project is generated by Capacitor and is **not** fully checked
in by the agent that scaffolded this package. Generate it and wire the two
native pieces:

```bash
# from the repo root
yarn install
yarn workspace @actual-app/ios build:web   # produces desktop-client/build
yarn workspace @actual-app/ios copy-web     # stages webdir
yarn workspace @actual-app/ios add:ios       # cap add ios  → creates ios/App
yarn workspace @actual-app/ios sync          # cap sync ios
```

Then, in Xcode (`yarn start:ios`):

1. **Add `COEPSchemeHandler.swift`** (already in
   `ios/App/App/`) to the `App` target if it isn't picked up automatically.

2. **Enable cross-origin isolation.** In `ios/App/App/Info.plist` add:

   ```xml
   <key>WKAppBoundDomains</key>
   <array>
     <string>localhost</string>
   </array>
   ```

   and wire the scheme handler in your `CAPBridgeViewController` subclass /
   `AppDelegate` so Capacitor's responses carry the COOP/COEP headers. The
   handler wraps the one Capacitor registers — see the doc comment at the top of
   `COEPSchemeHandler.swift` for the exact integration point.

   > **If you skip this:** the app still works. The web layer detects the
   > missing isolation and automatically switches to the slower
   > no-`SharedArrayBuffer` mode (`browser-preload.js`). On iOS there is only a
   > single WebView, so that fallback is safe.

3. **Icons & splash.** Generate the asset catalogs from a 1024×1024 source:

   ```bash
   npx @capacitor/assets generate --ios \
     --assetPath ../../desktop-client/public \
     --iconBackgroundColor '#5c3dbb' --splashBackgroundColor '#5c3dbb'
   ```

   (Provide a 1024px logo; the brand assets in `desktop-client/public` are 512px.)

## Daily development

```bash
yarn build:ios                 # from repo root: build web + stage + cap sync
yarn start:ios                 # open the project in Xcode
# or, run straight onto a Simulator/device:
yarn workspace @actual-app/ios run:ios
```

Verify in Safari Web Inspector (Develop ▸ Simulator ▸ Actual):

- `self.crossOriginIsolated === true` and `typeof SharedArrayBuffer === 'function'`
  (only when the COEP handler is wired; otherwise expect the fallback)
- the WASM module loads with `Content-Type: application/wasm`
- no CSP violations, a budget can be created, and data survives a relaunch

## App Store / TestFlight pipeline

Signing uses [`fastlane match`](https://docs.fastlane.tools/actions/match/)
(app-store certs in a private git repo) and an App Store Connect API key.
Configuration lives in `ios/App/fastlane/` (`Appfile`, `Matchfile`, `Fastfile`)
and `ios/App/Gemfile`.

```bash
cd packages/desktop-ios/ios/App
bundle install
bundle exec fastlane beta      # build + upload to TestFlight
bundle exec fastlane release   # build + upload to App Store (no auto-submit)
```

CI runs the same lanes via `.github/workflows/ios.yml` (manual
`workflow_dispatch`) on a `macos-latest` runner.

### Required secrets / one-time human steps

These need an Apple Developer account and **cannot** be done by an automated agent:

| Secret (GitHub `release` env)                    | What it is                                                          |
| ------------------------------------------------ | ------------------------------------------------------------------- |
| `APPLE_TEAM_ID`                                  | Apple Developer Team ID                                             |
| `ASC_KEY_ID`, `ASC_ISSUER_ID`, `ASC_KEY_CONTENT` | App Store Connect API key (`ASC_KEY_CONTENT` = base64 of the `.p8`) |
| `MATCH_GIT_URL`                                  | private git repo holding signing certs/profiles                     |
| `MATCH_PASSWORD`                                 | match encryption passphrase                                         |
| `MATCH_GIT_BASIC_AUTHORIZATION`                  | base64 `user:token` with read access to the match repo              |
| `ASC_TEAM_ID`                                    | (optional) App Store Connect team id, if your account has several   |

One-time, locally:

1. Register the bundle id `org.actualbudget.app` and create the app record in
   App Store Connect (plus the App Privacy questionnaire).
2. Seed signing assets: `cd ios/App && bundle exec fastlane match appstore`.
3. Add the secrets above to the repository's `release` environment.

The marketing version (`CFBundleShortVersionString`) is read from this package's
`package.json` `version` by the Fastfile, so keep it in lockstep with
`@actual-app/web`. The build number comes from the CI run number.
